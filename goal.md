# Цель проекта Deraft

## Краткая цель
Создать на C++ открытую, расширяемую и производительную воксельную sandbox-игру в духе Minecraft с Forge-подобной системой моддинга, позволяющей сообществу добавлять блоки, предметы, измерения, биомы, сущности, рецепты, команды, интерфейсы и правила игры без изменения ядра проекта.

## Что должно получиться
Deraft должен стать самостоятельной игрой и платформой для модов:

- Бесконечный или практически бесконечный процедурный воксельный мир на чанках.
- Система блоков, жидкостей, освещения, биомов и генерации ландшафта.
- Игровые режимы выживания, творчества и серверной песочницы.
- Клиент-серверная архитектура для одиночной игры, локального сервера и выделенных серверов.
- Рендеринг, пригодный для больших воксельных сцен, с поддержкой текстур-паков и моделей.
- Сохранения миров с версионированием и миграциями.
- Полноценный SDK для моддеров: API, документация, шаблон мода, события, реестры и инструменты сборки.

## Моддинг как ключевая цель
Forge-подобные возможности должны включать:

1. Загрузчик модов
   - Поиск модов в каталоге пользователя и сервера.
   - Чтение манифеста мода.
   - Проверка зависимостей, конфликтов и совместимости версий.
   - Раздельная загрузка client-only, server-only и universal модов.

2. API регистрации контента
   - Блоки, предметы, рецепты, сущности, биомы, измерения, команды, эффекты и теги.
   - Отложенная регистрация, чтобы моды могли безопасно ссылаться друг на друга.
   - Стабильные идентификаторы вида `namespace:name`.

3. Событийная система
   - События жизненного цикла игры и сервера.
   - События мира, чанков, блоков, сущностей, игрока, инвентаря, крафта и сетевого обмена.
   - Приоритеты обработчиков и возможность отменяемых событий там, где это безопасно.

4. Data-driven контент
   - Рецепты, лут-таблицы, теги, настройки генерации, модели и локализация должны быть описываемыми данными.
   - Моды должны иметь возможность поставлять ресурсы без перекомпиляции ядра.

5. Инструменты для моддеров
   - Шаблон минимального мода.
   - CLI для проверки манифеста и упаковки мода.
   - Документация по API и примеры типичных модов.

## Ограничения и юридическая чистота
- Не использовать исходный код Minecraft, Forge, Mojang/Microsoft или закрытых модов.
- Не копировать оригинальные текстуры, звуки, модели, торговые марки и уникальные визуальные элементы.
- Реализовывать собственные форматы, ассеты и названия.
- Ориентироваться на совместимость идей и удобство моддинга, а не на бинарную совместимость с Minecraft или Forge.

## Этапы разработки

### Этап 1: фундамент
- Настроить C++20-проект, сборку, тесты и базовую структуру каталогов.
- Реализовать базовые типы блоков, чанков и координат.
- Добавить простую генерацию мира и сохранение/загрузку.

### Этап 2: прототип игры
- Добавить рендеринг чанков, камеру, ввод и базовое взаимодействие с блоками.
- Реализовать инвентарь, предметы и простые рецепты.
- Создать минимальный игровой цикл клиента и сервера.

### Этап 3: основа моддинга
- Реализовать манифесты модов, загрузчик и проверку зависимостей.
- Добавить реестры и событийную шину.
- Сделать первый пример мода, добавляющий блок, предмет и рецепт.

### Этап 4: расширение API
- Добавить события мира, сущностей, игрока, сети и интерфейса.
- Поддержать data-driven ресурсы и теги.
- Зафиксировать политику версионирования API.

### Этап 5: стабильная платформа
- Улучшить производительность чанков, освещения, сетевого кода и сохранений.
- Подготовить документацию для игроков, серверов и моддеров.
- Создать набор regression-тестов для совместимости модов.
