# Руководство для агентов проекта Deraft

## Контекст проекта
Deraft — долгосрочный проект по созданию на C++ открытой воксельной sandbox-игры, вдохновлённой Minecraft, с модульной архитектурой и Forge-подобной системой моддинга.

Проект не должен копировать закрытые ассеты, торговые марки, исходный код или проприетарные форматы Minecraft. Разрешено воспроизводить общие игровые идеи жанра: воксельный мир, блоки, чанки, крафт, выживание, биомы, сущности и пользовательские моды.

## Основные принципы разработки
- Писать переносимый современный C++: целевой стандарт C++20 или новее.
- Разделять движок, игровую логику, контент и API моддинга.
- Держать ядро минимальным, расширяемым и хорошо документированным.
- Проектировать API модов как стабильный контракт, а не как набор внутренних хаков.
- Любая новая подсистема должна иметь понятные границы ответственности и тестируемую логику.
- Не добавлять зависимости без явного обоснования в документации или описании изменения.

## Предлагаемая архитектура
- `engine/` — платформенный слой, рендеринг, ввод, звук, файловая система, ресурсы.
- `core/` — воксельные структуры, чанки, генерация мира, блоки, предметы, ECS/сущности, сохранения.
- `game/` — стандартная реализация правил игры, ванильный контент Deraft и игровые режимы.
- `modding/` — загрузчик модов, API, события, реестры, жизненный цикл модов, совместимость версий.
- `tools/` — редакторы, конвертеры ассетов, генераторы данных, CLI-утилиты.
- `tests/` — модульные, интеграционные и регрессионные тесты.
- `docs/` — дизайн-документы, спецификации API, форматы файлов и руководства для моддеров.

## Forge-подобная система моддинга
Моддинг должен быть одной из ключевых возможностей проекта:

- Реестры для блоков, предметов, биомов, измерений, рецептов, сущностей, команд и эффектов.
- Событийная шина с фазами жизненного цикла: discovery, pre-init, init, post-init, server-start, world-load, gameplay и shutdown.
- Загрузка модов из отдельных пакетов с манифестом, зависимостями, версией API и списком ресурсов.
- Поддержка data-driven контента: JSON/TOML/YAML или другой явно выбранный формат для рецептов, лута, тегов, моделей и настроек.
- Изоляция модов от внутренних деталей движка там, где это возможно.
- Версионирование API моддинга и понятные ошибки при несовместимости.
- Возможность серверных, клиентских и универсальных модов.
- Документация каждого публичного интерфейса моддинга.

## Правила для изменений
- Перед крупными изменениями обновлять или создавать документ в `docs/` с описанием решения.
- Для игровой логики предпочитать детерминированные алгоритмы, пригодные для серверной симуляции.
- Не смешивать сетевой протокол, рендеринг и правила игры в одном модуле.
- Все форматы сохранений и сетевые структуры должны иметь версию.
- Для производительных участков кода добавлять бенчмарки или хотя бы описывать ожидаемые ограничения.
- Публичные API должны иметь примеры использования.

## Стиль документации
- Писать документацию на русском, если задача явно не требует английский.
- Для пользовательского API добавлять краткое назначение, пример и ограничения.
- Для спорных архитектурных решений фиксировать альтернативы и причины выбора.
